# Hailo NPU Investigation — Avis Birdfeeder Phase 8

**Date:** 2026-04-22
**Authors:** Kane Cruz-Walker, AI pair (Claude)
**Project:** Avis AI Birdfeeder (CS 450, SDSU Spring 2026)
**Status:** Investigation complete. Implementation plan adopted (Branches 1-5).
**Related PRs:** #38 (Hailo classifier benchmark), #46–#51 (Phase 8 preparation)

## TL;DR

During Phase 8 of the Avis project, deployment data showed that the system was dispatching empty-feeder scenes as confident bird observations (e.g., HOSP at 0.93 confidence on frames containing no bird). This investigation traced the root cause and evaluated whether the existing Hailo NPU pipeline could be salvaged as part of the fix.

Through seven systematic probes, we found three distinct issues: (1) the visual classifier's uncertainty signal cannot distinguish empty scenes from real birds because the LogReg head has no "no bird" class; (2) the Hailo-compiled EfficientNet classifier has input normalization and output dequantization bugs in the glue code that cause it to produce garbage predictions; and (3) the stock `yolov8s_h8l.hef` does not produce usable detections via our Python VDevice inference path, likely requiring Hailo's Tappas GStreamer pipeline or a custom-compiled HEF.

We decided to ship a CPU-based YOLO bird-presence gate (via ultralytics) for the immediate Phase 8 fix, while preserving all Hailo infrastructure for future restoration. The Hailo classifier fix is shallow and scheduled as Branch 3. A custom-compiled YOLO HEF is deferred as Branch 5 (post-report).

## Context

### The problem

On deployment days 2 and 3 (April 20-21, 2026), the `avis.service` systemd unit was running in production mode with the following configuration:

- `detection_mode: "fixed_crop"` (not using YOLO, due to concerns raised during Day 2)
- `hailo.enabled: false` (CPU-only inference path for both detection and classification)
- Motion threshold: 0.005 (permissive — constantly gathering)
- Confidence threshold: 0.20 for fusion dispatch

Under this configuration, `logs/observations.jsonl` accumulated records with the following pattern:

- Motion triggered → fixed crop of the feeder ROI sent to EfficientNet classifier
- Classifier returned confident species predictions (0.90+) even on visually empty feeder frames
- Fusion passed these to the notifier
- Many were dispatched via Pushover before the species-cooldown gate took effect

The baseline characterized 21 observation records from a 04:52-06:09 UTC window on 2026-04-22 that showed this pattern clearly: cam0 classified wood/ivy/orange backgrounds as HOSP at 0.93, while cam1 reported mixed WBNU/AMCR/HOSP classifications at 0.2-0.6 confidence on the same empty scenes.

This was a grade-blocking problem. An evaluation report premised on "our system correctly identifies feeder visitors" cannot withstand logs demonstrating that the system confidently identifies *empty feeders*.

### Why investigate rather than immediately rewrite

At the start of this session, the natural impulse was to propose a bird-presence gate and begin implementation. Instead, we chose to probe the existing pipeline first. This choice was driven by two considerations:

1. **The Hailo work on PRs #38 and #48 was hard-won.** PR #38 established a 6.3× NPU speedup for the EfficientNet classifier; PR #48 resolved ROUND_ROBIN scheduling issues. Ripping out this infrastructure without understanding why day-2 testing failed would likely recreate the same problems later.

2. **The right fix depends on what's actually broken.** A classifier-side fix (e.g., entropy gating) is 4 hours. A structural fix (bird-presence gate) is 1-2 days. A Hailo recompilation is 4-6 hours to a week depending on issues. Choosing the right fix required knowing which component was at fault.

Seven probes were run over ~2 hours. Each yielded decisive information. The aggregate cost was meaningful but the resulting implementation plan is concrete, scoped, and low-risk.

## Investigation Timeline

### Probe 1: Can classifier uncertainty gate empty-feeder noise?

**Hypothesis.** If the classifier's probability distribution is more uncertain (higher entropy, lower top1-top2 margin) on empty scenes than on real birds, we can add a lightweight uncertainty gate without any architectural changes.

**Method.** Ran the CPU-path `VisualClassifier` on 8 images with known ground-truth labels: 6 images from day-3 data plus 2 confirmed empty-feeder frames. Captured full top-5 probability distributions, entropy, and top1-top2 margin.

**Result.** The signal is inverted from what a gate would need. Empty feeders produced *lower* entropy and *higher* margin than most real bird images, because the classifier collapses confidently on scene-texture features it was never trained to recognize as "not a bird."

| Scene | Top Class | Confidence | Entropy | Margin |
|---|---|---|---|---|
| Clear crow | AMCR | 0.946 | 0.265 | 0.908 |
| Empty feeder | HOSP | 0.930 | 0.356 | 0.894 |
| Empty feeder | HOSP | 0.911 | 0.411 | 0.868 |
| Real HOSP | HOSP | 0.878 | 0.551 | 0.806 |
| Real HOFI (corner) | HOSP | 0.459 | 1.243 | 0.098 |

No entropy or margin threshold separates empty scenes from correctly-classified real birds.

**Interpretation.** This is actually a meaningful finding for the report: the failure mode is a concrete example of *distribution shift* in transfer learning. NABirds training photos frame birds as the dominant feature; our feeder crops frame birds as small fractions of a scene. The classifier learned "unified feature signal → peaked distribution" — not "bird-like signal → peaked distribution." Uncertainty-based gating cannot fix this. A dedicated bird-presence detector is required.

### Probe 2: Does the Hailo classifier produce predictions matching the CPU classifier?

**Hypothesis.** Re-enabling `hailo.enabled: true` should produce classification results within some tolerance of the CPU path on identical inputs.

**Method.** Enabled Hailo on the Pi, ran the classifier on 4 reference images (clear crow, real HOSP, HOFI corner, empty feeder) through both the CPU forward pass and the Hailo forward pass, and compared top-1 class and confidence.

**Result.** The Hailo classifier produced confident-looking but incorrect predictions across all 4 images, with top-1 confidences uniformly in the 0.14-0.16 range.

| Image | CPU top-1 | CPU conf | Hailo top-1 | Hailo conf | Δ |
|---|---|---|---|---|---|
| Clear crow | AMCR | 0.946 | AMCR | 0.150 | 0.797 |
| HOFI corner | HOSP | 0.459 | WBNU | 0.142 | 0.403 |
| Empty feeder | HOSP | 0.930 | DOWO | 0.155 | 0.852 |
| Real HOSP | HOSP | 0.878 | DOWO | 0.148 | 0.829 |

The uniformity of the Hailo output (~0.15 regardless of input) is the signature of a classifier being fed near-uniform feature vectors — i.e., the upstream Hailo EfficientNet extractor is producing garbage features, not that the LogReg head is broken.

**Interpretation.** The Hailo classifier path is not usable in its current state. The 0.15 floor observed during Day 2 testing was not caused by phone-screen images or environmental conditions; it was the Hailo extractor producing meaningless outputs regardless of input. See Finding 2 for root cause.

### Probe 3: Does `HailoDetector` initialize and invoke correctly from a standalone context?

**Hypothesis.** If PR #48's ROUND_ROBIN fix is complete, the YOLO detector should produce detections when called directly (outside the service).

**Method.** Ran `HailoDetector.detect()` on the clear-crow image via a bare VDevice (no shared ROUND_ROBIN VDevice).

**Result.** `HAILO_STREAM_NOT_ACTIVATED(72)` error immediately on `run_async()`.

**Interpretation.** `HailoDetector.open()` fallback path (when `shared_vdevice=None`) creates a plain `VDevice()` without the ROUND_ROBIN scheduling algorithm. The ROUND_ROBIN fix from PR #48 is only applied to `VisionCapture._create_shared_vdevice()`. This is not a bug in production (production uses the shared VDevice), but it means standalone diagnostic scripts must use the shared VDevice construction path.

### Probe 4: With the shared ROUND_ROBIN VDevice, does `HailoDetector` produce detections?

**Hypothesis.** Using the correct VDevice construction, the detector should run cleanly and produce YOLO detections on a canonical bird image.

**Method.** Created the shared VDevice via `_create_shared_vdevice()` and passed it to `HailoDetector`. Ran on the clear-crow image with permissive thresholds (score_threshold=0.05, min_bird_confidence=0.05).

**Result.** No errors. No bird detections. Two spurious `person` detections with low confidence.

**Interpretation.** The pipeline is alive but not producing correct output. This prompted deeper probing of the input and output format assumptions.

### Probes 5-6: Channel order and resize strategy

Ran the detector under RGB and BGR channel orders, then with stretched resize vs aspect-preserving letterbox. All four combinations produced either zero detections or the same spurious person detections. Channel order and resize strategy are not the issue.

### Probe 7: HEF metadata inspection, input format variants, and ultralytics ground-truth reference

**Method.** Three sub-probes:

1. **HEF metadata inspection** via `hef.get_input_vstream_infos()` and `hef.get_output_vstream_infos()` to see exactly what format the compiled model expects
2. **Input variant sweep** — tested raw uint8, BGR uint8, centered uint8, solid-color synthetic, bounded random noise, solid white, solid black
3. **Ultralytics reference** — installed ultralytics on the Pi, ran `YOLO('yolov8s.pt')` on the same canonical bird image and the same crow image, compared results

**Results.**

*HEF metadata for the classifier (`efficientnet_b0_avis_v2.hef`):*
- Input: `(224, 224, 3)`, UINT8, NHWC, quant_scale=0.003921568859368563 (≈1/255), quant_zero=0.0
- Output: `(1280,)`, UINT8, NC, quant_scale=0.014975594356656075, quant_zero=17.0

*HEF metadata for YOLO (`yolov8s_h8l.hef`):*
- Input: `(640, 640, 3)`, UINT8, NHWC, quant_scale=1.0, quant_zero=0.0
- Output: `yolov8_nms_postprocess`, reported as `(80, 5, 100)` FLOAT32, but HailoRT expects a 16320-byte buffer (HAILO_NMS_BY_CLASS compact format)

*Input variant sweep on canonical bird image:*

| Input variant | Bird detections | Other detections |
|---|---|---|
| Raw uint8 [0, 255] | 0 | None |
| BGR uint8 | 0 | None |
| Contiguous uint8 | 0 | None |
| Mean-subtracted uint8 | 0 | None |
| Solid sky blue | 0 | None |
| Bounded random noise | 0 | None |
| All white | 0 | None |
| All black | 0 | None |

*Ultralytics CPU reference on same images:*

| Image | Ultralytics CPU result |
|---|---|
| Canonical bird (unsplash) | 2 × class 14 (bird) @ 0.706, 0.656 |
| Clear crow (day-3 capture) | 1 × class 14 (bird) @ 0.852 + 2 low-conf misclassifications |

**Interpretation.** Three distinct findings:

1. **The Hailo classifier input/output format is mismatched in glue code** (Finding 2). The HEF expects raw uint8 pixels with internal 1/255 scaling; current code passes ImageNet-normalized floats reduced to uint8, which is double normalization. Output needs `(u8-17)*0.01497` dequantization, not `/255`.

2. **The stock YOLO HEF does not produce usable output via our Python inference path** (Finding 3). No input variant — including synthetic images that any functional object detector would respond to — produces any detections. The ultralytics reference confirms that the YOLOv8s weights and architecture are fully capable of detecting the same birds; the HEF or our invocation is at fault.

3. **Ultralytics CPU YOLO works correctly on our actual data.** Single-shot latency on Pi 5 is ~1000ms per frame, which is acceptable given our ~8 second motion-gated cycle interval.

## Key Findings

### Finding 1: Classifier uncertainty cannot gate empty-feeder noise

The `VisualClassifier` LogReg head has 19 output classes (one per SD species). It does not have a "no bird" class. When fed a feeder scene without a bird, EfficientNet features are dominated by wood/ivy/orange textures, which map to specific neighborhoods in the 1280-dim feature space. The LogReg head confidently picks whichever species is closest to that neighborhood — typically HOSP, whose NABirds training photos heavily featured neutral backgrounds.

Consequence: no entropy threshold, margin threshold, or confidence threshold can filter these out. Empty-feeder confidence is indistinguishable from real-bird confidence across the metrics we measured.

**Resolution path:** Add an upstream bird-presence gate before classification. This is Branch 2.

### Finding 2: Hailo classifier has input normalization and output dequantization bugs

The compiled HEF (`efficientnet_b0_avis_v2.hef`, from the April DFC build) is correct. The metadata explicitly declares UINT8 input with internal scale 1/255, meaning the hardware accepts raw pixel values [0, 255] and handles normalization on-chip. The output is UINT8 with quant parameters that require `(u8 - 17) * 0.01497` to recover float features.

The glue code violates both contracts:

- **Input side** (in `src/vision/classify.py.predict()`): passes `(frame * 255).clip(0, 255).astype(np.uint8)` where `frame` is already ImageNet-normalized to roughly [-2.1, 2.4]. Negative values clip to 0, values above 1.0 saturate to 255, resulting in a binary-clipped mangle of the input.
- **Output side** (in `src/vision/hailo_extractor.py.extract()`): returns `self._output_buffer.astype(np.float32) / 255.0`, treating the raw uint8 output as if it were a normalized value. The correct operation is `(uint8 - 17.0) * 0.01497559`.

**Resolution path:** Shallow fix in `classify.py` (pass raw uint8 RGB frame directly) and `hailo_extractor.py` (correct dequantization math). The input fix requires plumbing the pre-normalization frame through; the output fix is a single-line math correction. Estimated effort: 2 hours including tests and validation. This is Branch 3.

**Validation criterion:** Re-run Probe 2. Top-1 class agreement should be 100% across all 8 reference images. Top-1 confidence delta between CPU and Hailo should be < 0.05 (INT8 quantization noise).

### Finding 3: Stock `yolov8s_h8l.hef` does not produce usable detections

Ultralytics CPU YOLOv8s correctly detects the clear crow at 0.85 confidence and the canonical bird at 0.71 confidence. The Hailo-compiled stock HEF at `/usr/share/hailo-models/yolov8s_h8l.hef` produces zero detections across all 80 COCO classes on all input variants, including synthetic images that should trigger at least some response.

Several hypotheses were tested and ruled out: channel order, resize strategy, contiguity, input normalization variants, and the output buffer size assumption. None produced any meaningful detections.

The most likely root cause — based on `dpkg -L hailo-tappas-core` showing extensive C++ YOLO post-processing code at `/usr/include/hailo/tappas/detection/yolo_postprocess.{cpp,hpp}` — is that the stock HEF was compiled expecting consumption through Hailo's Tappas GStreamer pipeline, which handles preprocessing and post-processing in C++ outside the HEF itself. Our Python VDevice inference path misses these steps.

**Resolution path:** Two options. Either (a) build our own YOLOv8s HEF via the DFC toolchain with preprocessing embedded, which we've done before for EfficientNet and is technically feasible, or (b) accept CPU YOLO as the production path and defer NPU YOLO indefinitely.

We chose option (b) for the immediate Phase 8 fix and the report, with option (a) preserved as Branch 5 future work. Rationale in the next section.

## Architectural Decisions

### Decision 1: CPU YOLO via ultralytics for Branch 2

**Rationale:**

- **Empirically validated.** Ultralytics correctly detects our backyard birds at high confidence on the first try; no debugging required.
- **Latency acceptable.** 1000ms per inference on Pi 5 CPU versus an ~8 second motion-gated cycle. YOLO runs on a small fraction of frames (only motion-triggered). Cycle time impact: ~12% in worst case.
- **No HEF debugging debt.** Eliminates the risk of Phase 8 slipping into a multi-day Hailo side quest with uncertain outcomes.
- **Future optimization clear.** If latency becomes a problem, ultralytics has an ncnn export path for ARM-optimized inference at roughly 300ms.
- **Report narrative intact.** The Hailo NPU story is preserved via PR #38's 6.3× classifier speedup benchmark; Branch 3 restores the classifier to NPU execution. The YOLO pivot is documented as sound engineering judgment, not a capability gap.

### Decision 2: Preserve `detection_mode` A/B experiment

The original `detection_mode` field in `BirdObservation` supports two crop strategies for the classifier input: `fixed_crop` (static ROI) and `yolo` (detector bounding box with padding). The `ExperimentOrchestrator` rotates these every 30 minutes for A/B analysis.

With the bird-presence gate in place, this A/B comparison becomes genuinely meaningful. Both modes now gate correctly on bird presence, so the comparison isolates the impact of crop strategy alone: does a tight YOLO bbox help species identification by removing background clutter, or does it hurt by losing context (perch, scale, background vegetation that hints at species behavior)? No published benchmark answers this for backyard birds.

**New semantics for `detection_mode`:**

- `fixed_crop`: YOLO runs as gate; classifier sees the fixed ROI crop from `hardware.yaml: cameras.feeder_crop_cam0/1`
- `yolo`: YOLO runs as gate; classifier sees `detection.as_crop(raw_frame, padding=yolo_crop_padding)`

In both modes, if YOLO returns no bird, the observation is logged with `gate_reason="no_bird_detected"` and classification is skipped.

### Decision 3: Keep shared VDevice infrastructure

`VisionCapture._create_shared_vdevice()` and the `shared_vdevice` parameter plumbing through `HailoDetector` and `VisualClassifier._load_hailo()` all remain in place. The gating condition for VDevice creation changes from `detection_mode == "yolo" and hailo_yolo_hef` to simply `hailo_enabled`.

This enables:
- Branch 3 to restore Hailo classifier via the shared VDevice with no infrastructure work
- Branch 5 to plug in a custom-compiled `HailoYOLODetector` via the same shared VDevice
- Clear separation between "Hailo hardware available" (infrastructure concern) and "which models run on Hailo" (model-level decision)

**Prerequisites:**
- DFC Docker environment (established during April EfficientNet compilation)
- ONNX export from ultralytics: `model.export(format='onnx', imgsz=640, dynamic=False)`
- Calibration dataset: 64-128 images that look like what the production YOLO will see (feeder-angle crops, not generic COCO images)

**Expected work:**
- ONNX export: 30 min
- Calibration prep: 30 min
- DFC compilation: 30-60 min compute time
- Validation scripting: 30 min
- Production integration: 30 min (via existing `BirdDetector` protocol from Branch 2)

**Known risks:**
- On-chip NMS vs CPU NMS is an architectural decision. On-chip requires specific DFC `--end-node` and `--postprocess-config` flags; CPU NMS is simpler but slower. Recommend CPU NMS for first compilation to match what ultralytics does.
- YOLOv8 has Silu activations and specific concat layers that historically require DFC attention. The EfficientNet compilation's SE-block avgpool fix doesn't transfer.
- Hailo's DFC YOLOv8 example scripts in the model zoo are the best starting reference.

**Validation criterion:** Custom HEF detection confidence on canonical bird reference image within 5% of ultralytics CPU (0.71 ± 0.03). Same crow image should produce detection confidence within 5% of 0.85.

**Swapping mechanism:** `BirdDetector` protocol in Branch 2 has two implementations. Config backend toggle activates `HailoYOLODetector` instead of `CPUYOLODetector`. No changes to `VisionCapture`, `BirdAgent`, or the observation schema.

With the five-branch plan complete, the report supports three ablations:

### Probe 1: Classifier uncertainty distribution

```python
# probe_1_classifier_entropy.py
# Runs each image through VisualClassifier, reports full top-5 distribution,
# entropy, and top1-top2 margin. Tests whether uncertainty signal can gate
# empty-feeder noise.
import sys
from pathlib import Path
import numpy as np
import torch
from PIL import Image

sys.path.insert(0, "/mnt/data/avis-birdfeeder")
from src.vision.classify import VisualClassifier
from src.vision.preprocess import preprocess_frame

IMAGES = [
    ("/mnt/data/avis-birdfeeder/data/captures/images/20260422_003435_480214_cam0.png",
     "AMCR — clear crow"),
    ("/mnt/data/avis-birdfeeder/data/captures/images/20260422_002553_412983_cam0.png",
     "HOFI — top-left corner"),
    ("/mnt/data/avis-birdfeeder/data/captures/images/20260422_060818_401766_cam0.png",
     "EMPTY — reference noise"),
    ("/mnt/data/avis-birdfeeder/data/captures/images/20260421_215828_741788_cam0.png",
     "HOSP — real sparrow"),
]


def probe(classifier, img_path, ground_truth):
    p = Path(img_path)
    if not p.exists():
        return None
    raw = np.array(Image.open(p).convert("RGB"))
    frame = preprocess_frame(raw, width=224, height=224)
    tensor = torch.from_numpy(frame.astype(np.float32)).permute(2, 0, 1).unsqueeze(0).to(classifier.device)
    with torch.no_grad():
        features = classifier._extractor(tensor).cpu().numpy()
    scaled = classifier._scaler.transform(features)
    probs = classifier._clf.predict_proba(scaled)[0]
    top5_idx = probs.argsort()[-5:][::-1]
    entropy = float(-np.sum(probs * np.log(probs + 1e-12)))
    sorted_probs = np.sort(probs)[::-1]
    margin = float(sorted_probs[0] - sorted_probs[1])
    print(f"\n{p.name}")
    print(f"  truth: {ground_truth}")
    for rank, idx in enumerate(top5_idx, 1):
        code = classifier._label_map[int(idx)]
        print(f"  {rank}. {code:6s} {probs[idx]:.4f}")
    print(f"  entropy = {entropy:.3f}  margin = {margin:.3f}")


def main():
    classifier = VisualClassifier.from_config("configs/paths.yaml")
    classifier._load()
    for path, truth in IMAGES:
        probe(classifier, path, truth)


if __name__ == "__main__":
    main()
```

### Probe 2: CPU vs Hailo classifier parity

```python
# probe_2_hailo_classifier_parity.py
# Compares CPU and Hailo classifier predictions on the same images.
# Reveals input normalization and output dequantization bugs in Hailo glue code.
import sys
from pathlib import Path
import numpy as np
import torch
from PIL import Image

sys.path.insert(0, "/mnt/data/avis-birdfeeder")
from src.vision.classify import VisualClassifier
from src.vision.preprocess import preprocess_frame

IMAGES = [
    ("/mnt/data/avis-birdfeeder/data/captures/images/20260422_003435_480214_cam0.png",
     "AMCR — clear crow"),
    ("/mnt/data/avis-birdfeeder/data/captures/images/20260422_002553_412983_cam0.png",
     "HOFI — top-left corner"),
    ("/mnt/data/avis-birdfeeder/data/captures/images/20260422_060818_401766_cam0.png",
     "EMPTY — reference noise"),
    ("/mnt/data/avis-birdfeeder/data/captures/images/20260421_215828_741788_cam0.png",
     "HOSP — real sparrow"),
]


def cpu_probs(classifier, frame):
    tensor = torch.from_numpy(frame.astype(np.float32)).permute(2, 0, 1).unsqueeze(0).to(classifier.device)
    with torch.no_grad():
        features = classifier._extractor(tensor).cpu().numpy()
    scaled = classifier._scaler.transform(features)
    return classifier._clf.predict_proba(scaled)[0]


def hailo_probs(classifier, frame):
    frame_uint8 = (frame * 255).clip(0, 255).astype(np.uint8)
    features = classifier._hailo_extractor.extract(frame_uint8)
    scaled = classifier._scaler.transform(features)
    return classifier._clf.predict_proba(scaled)[0]


def main():
    classifier = VisualClassifier.from_config("configs/paths.yaml")
    classifier._load()
    classifier._load_hailo()
    for path, truth in IMAGES:
        p = Path(path)
        if not p.exists():
            continue
        raw = np.array(Image.open(p).convert("RGB"))
        frame = preprocess_frame(raw, width=224, height=224)
        pc = cpu_probs(classifier, frame)
        ph = hailo_probs(classifier, frame)
        cpu_top = int(pc.argmax())
        hailo_top = int(ph.argmax())
        print(f"{p.name}")
        print(f"  CPU  : {classifier._label_map[cpu_top]:6s} {pc[cpu_top]:.4f}")
```

### Probe 6: Stretched resize vs letterbox

```python
# probe_6_resize_strategy.py
# Tests stretched non-aspect-preserving resize vs YOLOv8 standard letterbox.
import sys
sys.path.insert(0, "/mnt/data/avis-birdfeeder")
import numpy as np
from PIL import Image
from src.vision.capture import _create_shared_vdevice
from src.vision.hailo_detector import HailoDetector


def letterbox(img, target_size=640, pad_value=114):
    h, w = img.shape[:2]
    scale = min(target_size / h, target_size / w)
    new_h, new_w = int(round(h * scale)), int(round(w * scale))
    resized = np.array(Image.fromarray(img).resize((new_w, new_h)))
    canvas = np.full((target_size, target_size, 3), pad_value, dtype=np.uint8)
    pad_top = (target_size - new_h) // 2
    pad_left = (target_size - new_w) // 2
    canvas[pad_top:pad_top + new_h, pad_left:pad_left + new_w] = resized
    return canvas


shared = _create_shared_vdevice()
detector = HailoDetector(
    hef_path="/usr/share/hailo-models/yolov8s_h8l.hef",
```
